# WordSnap 设计文档

> 原型图：[wordsnap-prototype-flow.png](/Users/weiyi/code/WordSnap/docs/wordsnap-prototype-flow.png)

## 1. 文档目的

本文档基于 `WordSnap` 当前原型图，输出一份可用于评审和实施的设计说明。文档同时约束技术方案需与 `WordFlow` 保持一致，便于后续直接复用现有工程经验、依赖体系和项目结构。

## 2. 产品定位

`WordSnap` 是一个围绕“拍照识词 -> 自动出题 -> 完成测试 -> 记忆分析 -> 错题巩固”闭环构建的英语学习应用。

相较于 `WordFlow` 的“无限流背词”模式，`WordSnap` 更强调从现实场景中的英文材料切入：

- 用户拍摄课本、试卷、读物或讲义中的英文内容
- 应用识别其中的目标单词
- 用户基于本次识别结果快速生成测试
- 系统根据答题结果形成记忆分析和错题复习入口

这意味着 `WordSnap` 的核心价值不是“词书先行”，而是“内容采集先行，学习闭环跟进”。

原型图展示了 8 个核心页面，底部还给出了清晰的主流程箭头：

`拍照识别 -> 生成考试 -> 完成考试 -> 记忆分析 -> 巩固复习`

### 3.1 页面清单

1. 首页
2. 拍照识别页
3. 识别结果页
4. 开始考试页
5. 答题页
6. 考试完成页
7. 单词记忆分析页
8. 错题详情页

### 3.2 视觉与交互风格

从原型来看，产品设计方向比较明确：

- 移动端优先，单手操作友好
- 首页突出单一主行动点，减少分心
- 识别和考试流程采用低干扰界面
- 蓝色承担主操作语义，红色承担“开始考试 / 强提醒 / 重新练错题”等动作语义
- 数据分析页使用图表和比例条做轻量可视化

## 4. 核心用户场景

### 4.1 课堂/课后快速练习

用户拍摄英语课文或单词表，系统自动提取单词并生成小测，用户在几分钟内完成一轮自测。

### 4.2 错题回收与强化

用户完成测试后，不需要重新翻找原文，直接从错题页把错误单词加入复习队列。

### 4.3 词本滚动积累

识别出的单词会被沉淀到单词本中，形成个人化学习资产，并持续参与后续分析和复习。

## 5. 功能范围

## 5.1 MVP 范围

- 拍照或相册导入图片
- OCR 识别文本并提取英文单词
- 展示识别结果并允许继续生成考试
- 根据识别结果或选定词本生成单词测试
- 支持单选/多选题
- 输出考试成绩
- 输出记忆分析结果
- 展示错题详情并支持加入复习
- 首页展示默认单词本与最近学习记录

## 5.2 后续增强范围

- 手动编辑识别结果
- 单词去重、词形还原、短语识别
- 拍照框选区域识别
- 错题自动进入复习计划
- 多词本管理与导入导出
- AI 辅助清洗 OCR 结果
- 与 `WordFlow` 的记忆算法进一步融合

## 6. 信息架构

- 底部导航
- 首页

- 单词本
- 统计

功能流页面作为独立路由推进：

- 拍照识别
- 识别结果
- 开始考试
- 答题
- 考试完成
- 记忆分析

## 7. 页面设计说明

## 7.1 首页

### 目标

承接用户进入应用后的第一次决策，并突出“拍照识别单词”这一核心入口。

### 主要模块

- 顶部标题与设置入口
- 主卡片：拍照识别单词
- 我的单词本摘要
- 最近学习列表
- 底部导航

### 关键交互

- 点击“开始识别”进入拍照识别页
- 点击设置进入设置页
- 点击单词本或最近学习可跳转到对应详情

## 7.2 拍照识别页

### 目标

让用户采集文本内容，并清楚感知系统当前正在识别什么。

### 主要模块

- 顶部导航栏：返回、标题、辅助操作
- 相机预览区域
- OCR 框选结果叠层
- 底部拍摄控制栏

### 关键交互

- 拍照
- 从相册导入
- 裁切/框选
- 进入识别结果页

### 设计要点

- OCR 框需要突出英文单词边界
- 允许用户快速确认“识别质量是否可接受”
- 对低质量图片需要给出重新拍摄建议

## 7.3 识别结果页

### 目标

将 OCR 输出转成用户可理解、可操作的“单词集合”。

### 主要模块

- 识别总数
- 单词标签区
- 重拍按钮
- 生成考试按钮

### 关键交互

- 返回重拍
- 直接使用识别结果生成考试

### 后续可扩展

- 支持删除错误识别项
- 支持合并重复词
- 支持只勾选部分单词参与考试

## 7.4 开始考试页

### 目标

在正式答题前，让用户配置测试范围和规则。

### 主要模块

- 单词范围
- 题目数量
- 每题选项数
- 是否允许多选
- 是否随机顺序
- 开始考试按钮

### 关键交互

- 修改考试参数
- 保存偏好设置
- 开始一轮测试

## 7.5 答题页

### 目标

以最小操作成本完成一题一题的词义判断。

### 主要模块

- 当前题号进度
- 单词本体
- 音标和发音入口
- 题干提示
- 选项网格
- 下一题按钮
- 收藏入口

### 关键交互

- 选择一个或多个答案
- 跳到下一题
- 收藏当前单词

### 设计要点

- 题面保持极简，保证注意力集中在单词和选项
- 多选模式要有明确提示
- 未选择时也允许下一题，便于区分“不确定”和“没答”

## 7.6 考试完成页

### 目标

给用户即时反馈，形成正向闭环。

### 主要模块

- 成绩主视觉
- 正确率
- 正确 / 错误 / 不知道 / 其他统计
- 查看错题按钮
- 查看分析按钮

### 关键交互

- 进入错题详情
- 进入分析页

## 7.7 单词记忆分析页

### 目标

把本次测试结果转成可理解的记忆状态。

### 主要模块

- 总体掌握情况环形图
- 记忆程度分布条形区
- 各层级数量与比例

### 建议的记忆层级

- 掌握
- 不熟悉
- 不确定
- 没学过
- 记不清

### 设计要点

- 图表服务于“下一步怎么学”，不只展示分数
- 各层级命名要能映射复习策略

## 7.8 错题详情页

### 目标

集中处理错误和不确定题目，作为复习入口。

### 主要模块

- 分类标签：全部 / 错误 / 不确定
- 错题卡片列表
- 每道题的正确答案
- 用户错误选择展示
- 加入复习按钮
- 重新练习错题按钮

### 关键交互

- 查看错误原因
- 将单词加入复习
- 重新生成错题练习

## 8. 业务流程设计

## 8.1 主流程

1. 用户从首页进入拍照识别
2. 用户拍照或选择图片
3. 系统执行 OCR，提取英文单词
4. 用户确认识别结果
5. 用户配置考试规则
6. 系统生成测试题
7. 用户完成答题
8. 系统计算成绩与记忆层级
9. 用户查看分析与错题
10. 错题进入复习队列或再次练习

## 8.2 数据沉淀流程

1. 识别结果进入本次会话缓存
2. 用户确认后，单词写入默认词本或指定词本
3. 答题记录写入学习记录
4. 学习记录更新记忆分析统计
5. 错题进入待复习集合

## 10. 技术设计

## 10.1 技术栈约束

`WordSnap` 技术栈必须与 `WordFlow` 保持一致，不另起一套体系。

### 基础技术栈

- 框架：Flutter 3.6.0+
- 语言：Dart
- 状态管理：Provider + SharedPreferences
- 数据可视化：fl_chart
- 网络请求：http + dio
- 本地存储：shared_preferences + path_provider
- 音频播放：audioplayers
- 文件处理：file_picker
- 动画效果：animated_text_kit
- 权限管理：permission_handler
- 应用信息：package_info_plus
- 外链打开：url_launcher
- Markdown 渲染：flutter_markdown
- 设备信息：device_info_plus
- 文件打开：open_filex

### 技术取舍说明

- 不引入新的状态管理框架，继续沿用 `WordFlow` 的 Provider 思路
- 不引入新的图表库，分析页继续使用 `fl_chart`
- 不引入新的存储方案，MVP 继续以本地存储为主
- 不默认要求自建后端，优先支持本地优先架构

## 10.3 页面与服务职责边界

保持与 `WordFlow` 一致的原则：

- 页面负责展示与交互编排
- Service 负责业务逻辑、接口调用、数据读写
- Model 负责结构化数据表达
- Core 层负责跨 feature 复用能力

## 10.4 本地存储设计

MVP 阶段建议全部走本地存储：

- 用户设置：`SharedPreferences`
- 最近识别会话：`SharedPreferences` 或本地 JSON
- 单词本与学习记录：`path_provider` + 本地文件
- 统计聚合结果：启动时按学习记录计算，或做轻量缓存

## 10.5 图表设计

分析页使用 `fl_chart`，图表建议如下：

- 环形图：总体掌握情况
- 横向比例条：记忆程度分布
- 后续可加折线图：近 7 天正确率 / 学习词数

## 10.6 权限设计

MVP 期至少涉及：

- 相机权限
- 相册/文件读取权限

统一通过 `permission_handler` 管理，并在首次触发时做场景化提示。

## 10.7 CI/CD 设计

这一点也需要与 `WordFlow` 保持一致，不只是应用代码结构一致。

### 目标

- 在 GitHub 上自动构建 Android APK
- 支持手动触发构建
- 支持通过版本标签自动生成 Release
- 尽量复用 `WordFlow` 已验证过的工作流思路

### 工作流设计

建议使用与 `WordFlow` 同风格的 GitHub Actions 工作流：

- 文件位置：`.github/workflows/build-and-release.yml`
- 触发方式：`workflow_dispatch`、`push tags`
- 基础步骤：
  - checkout
  - setup Java 17
  - setup Flutter stable
  - `flutter pub get`
  - `flutter build apk --release`
  - 上传 artifact
  - tag 触发时创建 GitHub Release 并附上 APK

### 针对 WordSnap 的适配点

由于当前 `WordSnap` 还是手工搭建的 Flutter 应用骨架，仓库里暂时没有完整的 `android/` 工程，因此 CI 需要先执行：

```bash
flutter create --platforms=android .
```

这样可以在不污染当前源码结构的前提下，让 GitHub Actions 在构建时自动补齐 Android 工程。

### 发布签名

与 `WordFlow` 一样，正式发布时建议通过 GitHub Secrets 注入签名信息：

- `SIGNING_KEY`
- `KEY_STORE_PASSWORD`
- `KEY_PASSWORD`
- `KEY_ALIAS`

如果未配置签名 secrets，则工作流仍应允许完成基础 APK 构建，用于内部验证。

## 11. 第三方接口依赖评审

这一章单独列出，方便评审时只看“哪些外部能力是必须引入的、哪些是可选的”。

## 11.1 必需依赖：OCR 识别接口

### 用途

将拍摄图片中的英文文本识别为结构化结果。

### 需要的返回数据

- 可确认的英文单词
- 大模型鉴别和补充后的词性/中文释义数组
- 置信度

### 评审重点

- 是否支持英文场景优化
- 是否支持返回坐标框
- 响应速度是否满足移动端体验
- 成本是否可控
- 是否允许前端直连，还是必须经由服务端代理

### 说明

如果采用端侧 OCR，则这一项可以不依赖第三方云接口；如果采用云 OCR，则应把 API Key 存储策略和调用链路单独评审。

## 11.2 高概率需要：词典/单词标准化接口

### 用途

把 OCR 识别出来的文本进一步转成可以出题的词汇数据。

### 可能承担的能力

- 去除无效词
- 词形还原
- 获取音标
- 获取或校准中文释义
- 获取或校准词性

### 为什么高概率需要

原型中的答题页和错题页已经展示出：

- 英文单词
- 音标
- 中文释义选项

如果这些数据不全部内置在本地词库中，就需要额外依赖外部词典或词汇查询接口。

### 评审重点

- 能否本地词库离线解决
- 是否必须实时联网
- 多义词如何处理
- OCR 误识别时如何兜底

## 11.3 可选依赖：AI 清洗与纠错接口

### 用途

对 OCR 结果进行补救性处理，提高可用率。

### 典型能力

- 从整段 OCR 文本中抽取“适合学习的英文单词”
- 自动合并大小写和复数形式
- 过滤冠词、介词等弱学习价值项
- 识别短语或专有名词

### 是否应在 MVP 引入

不建议作为 MVP 的强依赖。

建议顺序：

1. 先做 OCR + 本地规则过滤
2. 再观察真实识别质量
3. 如果错误率高，再考虑接入 AI 清洗能力

## 11.4 可选依赖：发音能力接口

### 用途

支持答题页中单词发音播放。

### 替代方案

- 使用系统 TTS
- 使用本地音频资源
- 使用第三方语音服务

### 建议

MVP 优先采用系统能力或本地资源，不建议一开始引入新的在线语音依赖。

## 11.5 当前建议的评审结论

### MVP 必审

- OCR 能力是否本地化还是第三方化
- 词典数据是否本地内置还是第三方查询

### 可后置

- AI 清洗
- 在线发音
- 云端账户与同步服务

## 12. 与 WordFlow 的复用关系

`WordSnap` 不应脱离 `WordFlow` 单独演化出一套陌生工程体系，而应尽量复用这些经验：

- `AppInitializer` 启动链路
- `AppTheme` 的全局主题管理
- `ResponsiveHelper` 的响应式适配
- `CompatiblePageRoute` 的统一导航体验
- `LearningDataService` 的学习记录思想
- `SettingsHelper` 的设置持久化思路
- `fl_chart` 图表使用经验

## 13. 风险与难点

### 13.1 OCR 质量不稳定

如果用户拍摄角度、光线或分辨率较差，识别结果可能大幅影响后续出题质量。

### 13.2 词典数据完整度不足

如果没有稳定的词义、音标和多义词处理能力，考试页体验会明显受损。

### 13.3 错题与记忆层级定义不统一

如果题目结果和记忆分析口径不一致，统计页会让用户感到混乱。

### 13.4 本地数据模型演进

从“单次会话练习”扩展到“长期词本积累”后，模型和存储会快速复杂化，需要提早把实体边界定好。

## 14. 实施建议

建议按 4 个阶段推进：

### 阶段一：闭环可用

- 完成拍照识别
- 完成识别结果确认
- 完成考试与成绩页

### 阶段二：沉淀学习资产

- 单词本写入
- 学习记录持久化
- 记忆分析落地

### 阶段三：巩固复习

- 错题分组
- 加入复习
- 重新练习错题

### 阶段四：质量增强

- OCR 结果编辑
- AI 清洗
- 词本管理
- 更完整的统计体系

## 15. 结论

从原型图来看，`WordSnap` 的产品方向已经足够清晰，适合做成一个“从真实文本进入学习”的轻量英语训练应用。

在技术上，最稳妥的方案不是重新发明一套工程体系，而是明确继承 `WordFlow` 的技术栈与程序架构，再围绕 OCR、词典数据和学习记录这三条主线推进。

如果下一步进入正式开发，建议优先完成两项评审：

1. OCR 能力采用本地方案还是第三方接口
2. 词义/音标数据采用本地词库还是第三方词典接口
