# 应用升级设计文档

## 1. 文档目的

本文档描述 WordFlow 当前的应用升级方案，目标是把现有实现抽象成一套可移植的设计，方便复用到其他应用中。

本文覆盖以下内容：

- 发布端如何产出可升级的安装包
- 客户端如何自动检查更新
- 客户端如何手动检查更新
- 客户端如何下载并触发安装
- 关键模块职责划分
- 迁移到其他应用时需要替换的配置项
- 当前方案的局限与优化建议

当前实现定位为：

- 自动检查更新
- 用户确认后下载
- 调用系统安装器完成安装
- Android 平台优先
- 基于 GitHub Release 分发 APK

这不是“静默自动升级”，而是“自动发现更新 + 半自动安装”。

## 2. 整体方案概览

WordFlow 的升级链路分为两部分：

1. 发布端生成新版本安装包，并上传到 GitHub Release。
2. 客户端定期或手动访问 Release API，发现新版本后下载 APK，并调起系统安装器。

高层流程如下：

1. 开发者修改 `pubspec.yaml` 中的版本号。
2. CI/CD 构建签名 APK。
3. CI/CD 创建 GitHub Release，并上传 `app-release.apk`。
4. 客户端启动后或回到前台时，自动检查最新 Release。
5. 用户在设置页也可以手动触发检查。
6. 若发现新版本，弹出更新对话框。
7. 用户确认后下载 APK。
8. 下载完成后调用系统安装器。
9. 用户在系统安装界面确认安装，完成升级。

## 3. 模块职责划分

当前实现主要由 4 个层次组成。

### 3.1 应用入口层

入口层负责在合适的生命周期触发更新检查。

对应代码：

- `lib/main.dart`

职责：

- 应用启动时初始化自动更新服务
- 应用首屏渲染后触发自动检查
- 应用从后台恢复到前台时再次触发检查

设计意图：

- 启动时检查，保证用户尽早发现新版本
- 恢复时检查，避免应用长时间驻留后台导致版本过旧
- 使用首帧后回调，避免更新检查阻塞首屏渲染

### 3.2 更新编排层

编排层负责决定“何时检查、如何区分自动和手动、何时弹窗、何时提示错误”。

对应代码：

- `lib/utils/auto_update_service.dart`

职责：

- 读取和保存上次检查时间
- 限制自动检查频率
- 区分自动检查和手动检查
- 调用版本检查接口
- 在有更新时展示更新对话框
- 在手动检查无更新时提示“已是最新版本”
- 在手动检查失败时提示错误

设计意图：

- 自动检查尽量安静，不打扰用户
- 手动检查必须给出明确反馈
- 用本地存储控制检查频率，避免频繁请求接口

### 3.3 版本信息层

版本信息层负责和服务端版本源交互，并完成版本比较。

对应代码：

- `lib/utils/settings_helper.dart`

职责：

- 获取本地应用版本号
- 请求 GitHub Release latest API
- 解析返回的版本信息
- 构造实际下载地址
- 比较当前版本和最新版本
- 格式化 Release Notes

设计意图：

- 将“版本源协议”和“版本比较逻辑”集中管理
- 降低 UI 层和下载层对 GitHub API 细节的耦合

### 3.4 下载与安装层

下载与安装层负责权限申请、文件下载、文件落地和触发系统安装。

对应代码：

- `lib/utils/update_manager.dart`

职责：

- 判断当前平台是否支持自动升级
- 检查和申请安装权限、存储权限
- 生成下载路径
- 下载 APK
- 支持取消下载
- 调用系统打开 APK 文件

设计意图：

- 将“更新包交付”与“版本检查”分离
- 方便未来替换下载器或安装方式

## 4. 发布端设计

客户端自动升级要成立，前提是发布端持续产出可被客户端识别的新版本。

### 4.1 版本来源

当前版本号来自：

- `pubspec.yaml`

示例：

```yaml
version: 1.0.10
```

客户端使用这个版本号作为本地当前版本。

### 4.2 Release 触发方式

当前 CI 工作流文件：

- `.github/workflows/build-and-release.yml`

触发条件：

- 推送符合 `v*.*.*` 的 Git tag
- 手动触发 `workflow_dispatch`

### 4.3 发布流程

CI 工作流的职责如下：

1. 拉取代码
2. 安装 Java 和 Flutter
3. 拉取依赖
4. 配置 Android 签名信息
5. 构建 release APK
6. 从 `pubspec.yaml` 提取版本号
7. 生成 Release Notes
8. 创建 GitHub Release
9. 上传 `app-release.apk`

### 4.4 客户端依赖的发布约束

当前客户端逻辑隐含依赖以下约束：

- GitHub 仓库固定为 `mikufoxxx/WordFlow`
- 最新版本通过 `/releases/latest` 获取
- Release tag 形如 `v1.0.10`
- 安装包文件名固定为 `app-release.apk`
- Release body 中建议包含“更新内容”和“安装说明”两个分段标题

如果迁移到其他应用，以上约束都需要根据目标项目调整。

## 5. 自动升级逻辑设计

这里的“自动升级”实际是“自动检查更新”。

### 5.1 触发时机

自动检查有两个入口：

1. 应用启动后的首帧回调
2. 应用从后台切回前台时

这样做的好处是：

- 不阻塞应用初始化
- 覆盖高频使用场景
- 不需要常驻后台任务

### 5.2 检查频率控制

当前实现通过本地存储记录 `last_update_check_time`，并约定：

- 自动检查间隔为 24 小时

判断规则：

- 如果从未检查过，允许检查
- 如果距离上次检查不足 24 小时，跳过检查
- 如果正在检查中，忽略重复请求

设计价值：

- 降低 GitHub API 调用频率
- 降低功耗和网络消耗
- 避免前后台切换时重复弹窗

### 5.3 自动检查的用户体验策略

自动检查遵循“少打扰”原则。

行为规则：

- 有更新时，展示更新弹窗
- 无更新时，不提示用户
- 检查失败时，不弹错误框
- 无更新但检查成功时，仍然刷新最后检查时间

这样设计的原因是：

- 自动检查是后台辅助能力，不应制造噪音
- 网络波动不应频繁影响用户主流程

## 6. 手动升级逻辑设计

### 6.1 触发入口

手动升级入口位于设置页的“检查更新”操作。

手动检查与自动检查复用同一套核心逻辑，但交互规则不同。

### 6.2 手动检查的用户体验策略

手动检查遵循“明确反馈”原则。

行为规则：

- 有更新时，弹出更新对话框
- 无更新时，明确提示“已是最新版本”
- 检查失败时，明确提示失败原因

这样设计的原因是：

- 手动操作代表用户主动期待结果
- 如果无反馈，用户会误以为按钮失效

### 6.3 自动与手动的差异总结

自动检查与手动检查的核心区别不在于版本判断逻辑，而在于交互反馈。

自动检查：

- 有更新才打扰用户
- 失败时静默

手动检查：

- 无更新也要提示
- 失败也要提示

这是移植到其他应用时必须保留的设计原则。

## 7. 版本检查逻辑设计

### 7.1 远端版本源

当前使用 GitHub 官方 API：

```text
https://api.github.com/repos/<owner>/<repo>/releases/latest
```

返回内容中主要使用以下字段：

- `tag_name`
- `body`
- `published_at`

### 7.2 本地版本获取

客户端通过 `package_info_plus` 获取当前安装版本号。

本地版本示例：

```text
1.0.10
```

远端 tag 示例：

```text
v1.0.11
```

解析时会先去掉前缀 `v`，得到：

```text
1.0.11
```

### 7.3 版本比较规则

当前实现按点分割版本号，再逐段转成整数比较。

例如：

- 当前版本：`1.0.9`
- 最新版本：`1.0.10`

比较过程：

1. 拆成数组
2. 长度不足时补零
3. 从左到右逐段比较
4. 任一段远端大于本地，则判定有更新

这是一种适合标准三段式版本号的轻量实现。

### 7.4 Release Notes 处理

Release body 会被用于更新弹窗展示。

当前逻辑会：

1. 优先提取“### 更新内容”和“### 安装说明”之间的部分
2. 如果找不到固定结构，再尝试识别其他标题
3. 如果仍然无法提取，则回退到默认文案
4. 为防止对话框过长，截断到 1000 字符以内

设计意义：

- 让 Release Notes 同时兼顾 GitHub 页面阅读和 App 内弹窗阅读

## 8. 下载与安装逻辑设计

### 8.1 平台限制

当前实现只支持 Android 自动下载和安装引导。

原因：

- Android 可以直接分发 APK
- iOS 不允许应用内自行下载并安装 IPA
- Windows、macOS、Web 需要完全不同的升级机制

因此迁移时应先明确目标平台。

### 8.2 下载地址生成

当前实现不是直接使用 Release API 返回的资产数组，而是根据 tag 规则拼接下载地址。

格式如下：

```text
http://git.techox.cc/https://github.com/<owner>/<repo>/releases/download/<tag>/app-release.apk
```

这里有两个关键点：

1. 安装包名被写死为 `app-release.apk`
2. 下载经过 `git.techox.cc` 镜像代理

如果迁移到其他应用，通常建议把以下内容做成配置项：

- 仓库名
- 资产文件名
- 是否启用镜像代理
- 镜像代理域名

### 8.3 权限申请

当前 Android 侧涉及以下权限：

- 网络权限
- 安装未知应用权限 `REQUEST_INSTALL_PACKAGES`
- Android 12 及以下的存储权限

当前逻辑：

- Android 13 及以上只申请安装权限
- Android 12 及以下申请存储权限和安装权限

### 8.4 本地文件落地

下载文件时，应用会：

1. 尝试使用外部存储目录
2. 在其下创建 `Download` 目录
3. 生成文件名 `wordflow-v<version>.apk`
4. 若已有同名旧文件，则先删除

这样做的好处是：

- 文件名带版本号，便于排查
- 避免旧文件干扰安装

### 8.5 下载过程

下载使用 `dio` 实现，支持：

- 进度回调
- 状态文案更新
- 用户取消下载

UI 上展示：

- 当前状态
- 进度条
- 百分比

### 8.6 触发安装

下载完成后，应用调用 `open_filex` 打开 APK 文件。

本质上并不是应用自己执行安装，而是把 APK 交给 Android 系统安装器。

后续流程由系统接管：

1. 系统弹出安装界面
2. 用户确认安装
3. 系统完成覆盖安装

## 9. Android 侧配置要求

若要在其他 Android 应用中复用该方案，至少需要以下配置。

### 9.1 Manifest 权限

需要关注的权限包括：

- `android.permission.INTERNET`
- `android.permission.ACCESS_NETWORK_STATE`
- `android.permission.REQUEST_INSTALL_PACKAGES`
- Android 低版本存储权限

### 9.2 FileProvider

当前项目在 `AndroidManifest.xml` 中配置了 `FileProvider`。

即使当前 `open_filex` 已经代为处理大多数细节，保留这类文件共享能力依然是比较稳妥的做法，尤其是在替换安装库或自行封装安装逻辑时。

### 9.3 签名一致性

升级安装要成功，前提是：

- 新版本 APK 与旧版本 APK 使用同一签名证书

因此发布端必须稳定维护签名密钥，不能随意更换。

## 10. 当前 UI 交互设计

更新弹窗承担以下职责：

- 展示当前版本
- 展示最新版本
- 展示更新说明
- 提供“稍后更新”入口
- 提供“立即更新”入口
- 下载时展示进度
- 下载完成后提示用户按系统引导完成安装

建议移植时保留以下交互原则：

- 默认不强制更新
- 对话框内容简洁
- 下载进度可视化
- 下载失败要有明确提示
- 自动检查与手动检查的反馈规则要区分

## 11. 失败场景与兜底策略

一个可移植的升级方案，必须明确失败场景。

### 11.1 版本检查失败

可能原因：

- 无网络
- GitHub API 超时
- API 限流
- 仓库地址错误
- Release 不存在

当前策略：

- 自动检查失败时静默
- 手动检查失败时提示错误

### 11.2 版本比较失败

可能原因：

- tag 格式不规范
- 版本号包含无法解析的字符

建议：

- 统一版本格式，例如 `v1.2.3`
- 对解析异常增加日志与保护

### 11.3 下载失败

可能原因：

- 下载地址失效
- 代理不可用
- 存储权限不足
- 设备空间不足

当前策略：

- 下载失败后恢复按钮状态
- 通过 `SnackBar` 提示失败

### 11.4 安装失败

可能原因：

- 用户未授予安装未知应用权限
- APK 文件损坏
- 签名不一致
- 安装包与设备架构不兼容

建议：

- 对安装失败增加更明确的错误文案
- 提供“重新下载”或“复制下载路径”的辅助能力

## 13. 推荐的可移植抽象

为了方便在其他应用中复用，建议把当前实现进一步抽象成以下接口。

### 13.1 UpdateSource

负责提供“最新版本信息”。

建议接口能力：

- 获取最新版本
- 获取发布说明
- 获取下载地址

可替换实现：

- GitHub Releases
- Gitee Releases
- 自建更新服务
- 企业内网文件服务

### 13.2 VersionComparator

负责比较本地版本与远端版本。

建议接口能力：

- 标准三段式比较
- 预发布版本处理
- 版本格式校验

### 13.3 UpdateScheduler

负责决定何时自动检查。

建议接口能力：

- 应用启动检查
- 前后台切换检查
- 固定时间间隔检查
- 防重入控制

### 13.4 UpdateInstaller

负责下载并引导安装。

建议接口能力：

- 权限检查
- 下载进度通知
- 下载取消
- 安装触发

### 13.5 UpdatePresenter

负责所有更新相关 UI。

建议接口能力：

- 发现新版本弹窗
- 无更新提示
- 错误提示
- 下载进度展示

## 14. 建议保留的设计原则

迁移时，建议优先保留以下原则，而不是机械复制当前代码。

1. 自动检查和手动检查复用同一核心逻辑。
2. 自动检查安静，手动检查有反馈。
3. 版本源、比较器、下载器、UI 编排分层解耦。
4. 发布端资产命名规则必须稳定。
5. 升级安装链路必须与签名策略绑定考虑。
6. 平台差异要在下载与安装层隔离。

## 15. 当前方案的局限

WordFlow 当前实现可以工作，但从可移植性和稳健性看，还有一些明显限制。

### 15.1 只支持 Android

当前安装流程直接围绕 APK 构建，不适用于 iOS、桌面端和 Web。

### 15.2 下载地址拼接较硬编码

当前不是从 Release 资产列表动态选择文件，而是直接拼接固定文件名。

影响：

- 更换资产命名就会失效
- 多架构包或多渠道包不易扩展

### 15.3 代理地址使用 HTTP

当前下载代理为明文 HTTP，存在安全和可靠性风险。

### 15.4 错误信息传递不够完整

`checkForUpdates()` 某些失败场景会返回 `hasUpdate: false`，但调用层未总是区分“真的无更新”和“检查失败”。

### 15.5 强依赖 GitHub Release 文本结构

Release Notes 展示效果依赖固定标题结构，不够鲁棒。

## 16. 移植时的优化建议

如果你准备把这套方案真正迁移到别的应用，建议在移植时顺手做下面几项增强。

1. 把仓库名、资产名、代理地址抽成配置项。
2. 直接解析 Release `assets` 列表，不再硬编码 `app-release.apk`。
3. 使用 HTTPS 下载源。
4. 在更新检查结果中显式区分“无更新”和“检查失败”。
5. 为 Android 以外平台预留独立实现。
6. 把更新弹窗与下载器解耦，便于替换 UI 框架。
7. 增加埋点或日志，方便分析升级成功率。

## 17. 适合直接复用的迁移步骤

如果你要在另一个 Flutter Android 应用里快速复用，建议按下面顺序实施。

1. 新建 `UpdateSource`、`UpdateScheduler`、`UpdateInstaller`、`UpdatePresenter` 四层结构。
2. 接入 `package_info_plus` 获取本地版本。
3. 接入远端 Release API。
4. 实现三段式版本比较。
5. 在应用启动后和前台恢复时接入自动检查。
6. 在设置页接入手动检查按钮。
7. 接入 `dio` 下载 APK。
8. 配置 Android 安装权限和 `FileProvider`。
9. 配置签名一致的 CI/CD Release 流程。
10. 用真机验证以下场景：无更新、有更新、下载失败、权限拒绝、安装成功。

## 18. 结论

WordFlow 的升级设计本质上是一套面向 Android 分发场景的轻量更新方案：

- 发布端用 GitHub Release 承载安装包
- 客户端用 GitHub API 判断是否有新版本
- 自动检查负责“发现更新”
- 手动检查负责“给出明确反馈”
- 下载器负责“拉取 APK 并引导系统安装”

如果要移植到其他应用，最值得复用的并不是某个具体类，而是这套分层思路：

- 生命周期触发
- 本地节流
- 远端版本解析
- UI 反馈分级
- 下载与安装解耦
- 发布端与客户端约束一致

只要保留这些核心原则，再替换掉仓库地址、安装包格式和平台适配层，这套方案就能比较顺利地迁移到其他项目中。
